# aurora-aml
Autonomous AML intelligence system for Australian financial transactions. Ensemble ML + SHAP explainability + local LLM narrative generation via Ollama.

Aurora: Autonomous AML Intelligence | Detect. Explain. Narrate. Locally.

Aurora is an autonomous anti-money laundering intelligence system built for Australian financial institutions. It combines ensemble machine learning, SHAP-based explainability, and local large language model narrative generation to produce compliance-ready Suspicious Transaction Report narratives, entirely on-premises, with no data leaving your infrastructure.
Built on the NVIDIA DGX Spark (GB10, CUDA 13.0). Designed for the Australian regulatory context (AUSTRAC).

What Aurora Does
A compliance analyst reviewing flagged transactions needs three things: a risk score, a reason, and a narrative they can submit. Aurora produces all three automatically.
For each flagged transaction, Aurora:

Assigns an ensemble risk score (LightGBM + Isolation Forest)
Computes SHAP feature contributions identifying which behavioral signals drove the alert
Generates a plain-English STR narrative via a local Ollama LLM (llama3.1:8b)
Displays everything in an interactive Streamlit dashboard

No cloud API calls. No data transmission. Everything runs on local hardware.

AUSTRAC Typologies Detected
Aurora is trained to recognize six behavioral typologies consistent with AUSTRAC's financial crime guidance:

Structuring -- transactions deliberately kept below the AUD 10,000 IFTI reporting threshold
Smurfing -- high-frequency low-value transactions designed to aggregate without triggering detection
Rapid Movement -- unusually high transaction velocity combined with large amounts
Dormant Reactivation -- long-inactive accounts suddenly processing significant transactions
Third-Party Layering -- international transfers with elevated frequency suggesting layering behavior
Crypto Gateway -- large international SWIFT transfers consistent with fiat-to-crypto conversion

Stage-by-Stage Build
Aurora was built in five stages:
Stage 1 -- Environment and synthetic data
Python virtual environment setup. Synthetic Australian transaction dataset generation using Faker (en_AU locale), with six AUSTRAC typologies embedded as behavioral patterns across 5,000 transactions.
Stage 2 -- Feature engineering
Regulatory knowledge translated into signal. Features include threshold proximity (distance from AUD 10,000), velocity-amount interaction, dormancy flags, off-hours indicators, and a composite rule-based risk score. Top correlations with ground truth: composite risk score (0.873), high velocity (0.729), velocity-amount interaction (0.728).
Stage 3 -- Ensemble model training
LightGBM trained with class-weight balancing on 19 features. Isolation Forest trained unsupervised as a complementary anomaly detector. Ensemble weighting: 70/30. Final AUC-ROC: 0.999.
Stage 4 -- SHAP explainability and narrative generation
SHAP TreeExplainer computes per-transaction feature contributions. Top four SHAP drivers are passed as structured context to a local Ollama LLM, which generates a 3-sentence STR narrative in plain compliance English.
Stage 5 -- Streamlit dashboard
Interactive dashboard with summary metrics, typology distribution chart, color-coded risk table, per-transaction SHAP chart, and live narrative generation button. Dark theme with teal and gold accents.

Key Design Decisions
Why local LLM?
Financial transaction data is sensitive. Sending it to a cloud API for narrative generation creates regulatory and privacy risk. Aurora uses Ollama to run inference entirely on-premises. The DGX Spark's GB10 chip handles llama3.1:8b comfortably with sub-5-second response times.
Why SHAP before narrative?
Raw model scores are not sufficient for compliance. A compliance officer filing an STR needs to articulate why a transaction was suspicious, not just that it was flagged. SHAP decomposes the model decision into interpretable contributors, which then become structured inputs to the LLM prompt, grounding the narrative in actual model reasoning rather than hallucinated justification.
Why ensemble?
LightGBM is a supervised classifier: it learns from labeled typology patterns. Isolation Forest is unsupervised: it detects statistical anomalies regardless of known patterns. Together they catch both known typologies and novel behavioral outliers not present in training data.

Limitations and Honest Caveats

The current dataset is synthetic. Real-world performance will differ and requires validation against actual transaction data before any production deployment.
The AUC-ROC of 1.000 on the synthetic test set reflects the structured nature of the generator, not real-world complexity. Real transaction data will produce lower scores. This is expected and healthy.
LLM narratives should be reviewed by a qualified compliance officer before submission. Aurora assists human judgment; it does not replace it.
This system is a research prototype. It has not been reviewed or certified for use in regulated financial environments.
